El problema: "Funciona en mi maquina"

Sin contenedores

  • "Funciona en mi maquina" ...pero no en produccion
  • Dependencias conflictivas entre proyectos
  • Deployments inconsistentes
  • Onboarding lento (instalar todo desde cero)

Con contenedores

  • Mismo entorno en dev, staging y produccion
  • Dependencias aisladas entre proyectos
  • Imagen inmutable y versionada
  • Onboarding en minutos: docker compose up

Interview Tip

"Cual es la diferencia entre VM y container?"

Pregunta: cuantos han tenido el problema de instalar algo y que no funcione? Ese es el problema que Docker resuelve. INTERVIEW TIP (Q3): "Cual es la diferencia entre virtualizacion y containerization?". VMs virtualizan el HARDWARE (hipervisor + SO completo por VM, GB, boot lento). Containers virtualizan el OS (comparten kernel, namespaces, MB, boot en ms). TRAMPA comun: decir "Docker es un VM ligero" - NO. No hay hipervisor, no hay guest OS. Follow-up tricky: "Si containers son mas ligeros, por que usar VMs?" (Aislamiento mas fuerte, correr diferentes OSes, apps legacy, seguridad).

VMs vs Contenedores

La diferencia clave: VMs virtualizan HARDWARE (GB), Contenedores virtualizan el OS (MB)

Interview Tip

"Cuál es la diferencia entre VMs y contenedores?"

La diferencia clave: VMs incluyen un SO completo por instancia (GBs). Contenedores comparten el kernel del host (MBs). Un contenedor arranca en milisegundos vs minutos para una VM. TRAMPA: decir "Docker es una VM ligera". No lo es - comparten el kernel del host, no hay hipervisor.

Namespaces y Cgroups: la magia por debajo

Namespaces — Aislamiento

Cada container ve su propio "mundo"

PID — Procesos propios

Network — Red propia

Mount — Filesystem propio

UTS — Hostname propio

IPC — Memoria compartida

User — Usuarios propios

Cgroups — Recursos

Limitamos cuánto puede usar cada container

  • CPU — Límite de uso de procesador
  • Memoria — RAM máxima que puede consumir
  • I/O — Ancho de banda de disco
  • Network — Ancho de banda de red
  • PIDs — Máximo de procesos

Sin cgroups, un container podría devorar todos los recursos del host

Interview Tip

"Qué feature del kernel Linux aísla y limita los containers?"

Namespaces crean vistas aisladas del sistema. PID namespace = el container cree que su proceso init tiene PID 1. Network namespace = el container tiene su propia IP. Cgroups limita: si pones memory=256m, el container solo usa 256MB de RAM aunque intente usar más.

Imagen vs Contenedor

Imagen = la receta

  • Inmutable, versionada con tags
  • Plantilla de solo lectura
  • Vive en un registry (Docker Hub)

Contenedor = el lunchbox

  • Instancia viva de una imagen
  • Proceso, red y storage propios
  • Puedes tener N de la misma imagen

Imagen = clase  •  Contenedor = instancia

Interview Tip

"Cual es la diferencia entre imagen y contenedor?"

La analogia de programacion orientada a objetos: la imagen es la clase, el contenedor es la instancia. O la receta (imagen) vs el pastel ya horneado (contenedor). INTERVIEW TIP (Q1): "Cual es la diferencia entre Docker image y Docker container?". Imagen = clase, blueprint read-only con tu codigo y dependencias. Contenedor = instancia de la imagen, es lo que esta corriendo. Puedes tener N contenedores de 1 imagen. TRAMPA: candidatos dicen "el contenedor ES la imagen" o confunden layers con imagenes. La imagen se construye de layers, el contenedor es lo que corre. Follow-up tricky: "Puede un contenedor cambiar su imagen?" (No - la imagen es inmutable. Construyes una nueva imagen y recreas el contenedor).

Dockerfile: La receta


FROM python:3.12-slim         # Imagen base

WORKDIR /app                  # Directorio de trabajo
COPY requirements.txt .       # Copiar dependencias
RUN pip install -r requirements.txt  # Instalar

COPY . .                      # Copiar codigo fuente
EXPOSE 8000                   # Puerto expuesto
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

docker build -t mi-app:v1 . && docker run -p 8000:8000 mi-app:v1

Interview Tip

"Cual es la diferencia entre CMD y ENTRYPOINT?"

Explica cada instruccion: FROM=base, WORKDIR=donde trabajamos, COPY=traer archivos, RUN=ejecutar al construir, EXPOSE=documentar puerto, CMD=que corre el contenedor. HEALTHCHECK es crucial para K8s despues. INTERVIEW TIP (Q2): "Cual es la diferencia entre CMD y ENTRYPOINT en un Dockerfile?". CMD = default command, facil de override. ENTRYPOINT = fixed command que siempre corre. Best practice: ENTRYPOINT para el main executable, CMD para default flags. TRAMPA comun: decir "son lo mismo". No lo son - `docker run image ls` reemplaza CMD pero se agrega a ENTRYPOINT. Follow-up: "Que pasa si tienes multiples CMD?" (Solo gana el ultimo).

Image Layers: cómo se construye una imagen

Cada instrucción del Dockerfile = 1 layer

  • Los layers son inmutables y compartidos
  • Se apilan uno encima del otro (Union Filesystem)
  • Docker solo guarda los layers que cambian
  • Varios contenedores comparten los mismos layers

# Ver los layers de una imagen
docker image history ubuntu
Cada capa es read-only. Cuando construyes un contenedor, Docker agrega una capa writable encima. Si cambias el código fuente, solo esa capa se reconstruye.

Build Cache en acción


$ docker build -t mi-app:v1 .

Step 1/6 : FROM ubuntu
 ---> bb0eaf4eee00
Step 2/6 : RUN apt-get update
 ---> Using cache          ← Cache HIT ✅
 ---> e09e593ec730
Step 3/6 : RUN apt-get install -y python python-pip
 ---> Using cache          ← Cache HIT ✅
 ---> a1b2c3d4e5f6
Step 4/6 : COPY app.py /opt/source-code
 ---> e9944225690a        ← Cache MISS ⚠️
Step 5/6 : RUN pip install flask
 ---> Running               ← Se ejecuta de nuevo
Step 6/6 : ENTRYPOINT ["flask", "run"]
 ---> f1a2b3c4d5e6

Successfully tagged mi-app:v1

Si solo cambias app.py, solo se reconstruyen los pasos 4, 5 y 6

Docker compara cada instrucción del Dockerfile con el cache. Si la instrucción y sus inputs son iguales, usa el cache. Si cambias algo, todos los steps posteriores se rebuild.

Cache Busting: cómo forzar la reconstrucción

Qué invalida el cache

  • Cualquier instrucción que cambie
  • Archivos copiados con ADD o COPY que cambien
  • El orden de las instrucciones importa
  • Variables de entorno que cambien

Best practices para el cache

  • Instrucciones que menos cambian van primero
  • Dependencias arriba, código abajo
  • .dockerignore para excluir archivos innecesarios
  • Poner COPY de código al final

Orden óptimo: FROM → RUN apt-get → COPY requirements.txt → RUN pip install → COPY código → CMD

Interview Tip

"Por qué tu build de Docker está lento?"

Si tienes `COPY . .` antes de las dependencias, cada vez que cambies código se rebuild todo. El orden correcto: primero las capas que cambian menos (sistema, dependencias), después el código.

Multi-stage Builds: imágenes más pequeñas


# ===== Build stage: instala todas las dependencias =====
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

# ===== Production stage: solo runtime =====
FROM python:3.12-slim
WORKDIR /app
# Copia SOLO lo necesario para correr
COPY --from=builder /usr/local/lib/python3.12/site-packages ./site-packages/
COPY --from=builder /app ./app/
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
requirements.txt  →  fastapi uvicorn[standard] pydantic

SIN multi-stage

~900 MB

python:3.12 + pip + dev libs

CON multi-stage

~180 MB

solo runtime + deps prod

Multi-stage copia SOLO el runtime de Python + dependencias de producción — descarta pip, dev libs y archivos innecesarios

Interview Tip

"Cómo optimizas el tamaño de una imagen Docker en producción?"

Multi-stage con Python/FastAPI: el build stage instala TODO (fastapi, uvicorn, pydantic, más las dependencias transitivas de cada uno). El prod stage copia solo site-packages/ + app/. El resultado es ~180MB vs ~900MB. El mismo patrón aplica para Django, Flask, FastAPI, Starlette, etc. Si la app usa dependencias con binarios C compilados (Pillow, numpy, lxml), el prod stage debe copiar /usr/local/lib/ completas.

Kubernetes

Orquestacion de contenedores a escala

Por que Kubernetes?

Docker solo

  • Si un contenedor muere — se queda muerto
  • Escalar manualmente (crear mas contenedores a mano)
  • Sin balanceo de carga automatico
  • Actualizar sin downtime — muy dificil
  • Si el servidor se cae, todo se cae

Con Kubernetes

  • Auto-healing: reemplaza pods muertos
  • Auto-scaling: escala segun demanda
  • Service discovery + load balancing
  • Rolling updates con rollback instantaneo
  • Si un nodo se cae, K8s reubica los pods
Docker es genial para una app en un servidor. Pero que pasa cuando tienes 50 apps en 10 servidores y uno se cae? Kubernetes resuelve eso.

Arquitectura de un Cluster

Flujo interactivo de comunicación entre los componentes del Control Plane y los Worker Nodes:

1. Solicitud (kubectl) El administrador envía un manifiesto YAML al API Server.
2. Almacenamiento (etcd) El API Server valida la orden y persiste el estado en la base de datos etcd.
3. Asignación (Scheduler) El Scheduler identifica nodos viables y asigna el Pod a un Worker Node.
4. Ejecución (kubelet) El kubelet del nodo recibe la asignación y ordena levantar los contenedores.
El control plane es el cerebro. El API server es la puerta. etcd guarda todo (key-value store). El scheduler decide donde van las cosas. Los workers ejecutan los pods reales. Fuente oficial: kubernetes.io/docs/concepts/architecture/. INTERVIEW TIP: "Explica la arquitectura de un cluster K8s". Control plane: kube-apiserver, etcd, kube-scheduler, kube-controller-manager, cloud-controller-manager. Worker nodes: kubelet, kube-proxy, container runtime. TRAMPA: confundir kubelet (agente en el nodo) con kubectl (el CLI que tu usas). Tambien: decir que el control plane corre containers (no - solo el API server interactua, los containers corren en workers). Follow-up tricky: "Que pasa si etcd se cae?" (Cluster muerto - no hay estado, no hay scheduling. Hay que restaurar del backup). Esta animación de DevOpsCube resume visualmente el ciclo. Al ejecutar kubectl apply, el API Server recibe la petición, etcd guarda la verdad absoluta del cluster, el Scheduler decide en qué nodo cabe el Pod según recursos disponibles, y el kubelet en dicho nodo es quien levanta los contenedores en containerd.

Componentes clave

Control Plane

API Server Puerta de entrada del cluster. Recibe comandos desde kubectl.
etcd Base de datos clave-valor. Almacena el estado y configuración de todo el cluster.
Scheduler Asigna nodos disponibles para ejecutar los pods recién creados.
Controllers Reguladores que comparan el estado deseado contra el real y corrigen diferencias.

Worker Nodes

kubelet Agente en cada nodo que recibe órdenes del Control Plane y gestiona los pods locales.
kube-proxy Administrador de red en el nodo. Mantiene reglas de red y balanceo de carga interno.
container runtime Motor que ejecuta contenedores dentro de los pods (como containerd o CRI-O).
Si un nodo se cae, Kubernetes reubica los pods en otro nodo automáticamente.
Los componentes trabajan juntos. El API server recibe tus comandos, el scheduler decide donde van los pods, el kubelet los ejecuta y los controllers vigilan que todo siga como debe.

Objetos de K8s

Pods, Deployments, Services

Pods

Analogia: roommates compartiendo apartamento

  • 1 o mas contenedores por pod
  • Comparten misma IP y almacenamiento
  • Efimeros: nacen y mueren
  • En produccion: usar Deployments

apiVersion: v1
kind: Pod
metadata: { name: mi-app }
spec:
  containers:
  - { name: app, image: mi-app:v1 }
  - { name: logger, image: logger:v1 }

Interview Tip

"Diferencia entre Pod, Deployment y StatefulSet?"

Un Pod es como un apartamento donde viven 1 o mas roommates (contenedores). Comparten todo (red, storage). Son efimeros. Por eso usamos Deployments para gestionarlos. INTERVIEW TIP (Q6): "Cual es la diferencia entre Pod, Deployment y StatefulSet?". Pod = unit de scheduling (1+ containers, shared net, ephemeral). Deployment = manages ReplicaSets, rolling updates, para apps stateless. StatefulSet = stable network IDs, persistent storage per pod, ordered scaling (db-0, db-1, db-2), para stateful apps como databases. TRAMPA: "Deployment = Pod" - NO, Deployment crea ReplicaSets que crean Pods. Tambien: candidatos ponen databases en Deployments. Follow-up: "Puedes usar Deployment para una database?" (Tecnicamente si, pero pierdes data al reschedule. Usa StatefulSet + PVC).

Deployments

Analogia: un gerente que contrata/despide

  • Define cuantos pods quieres (replicas)
  • Si un pod muere — crea uno nuevo
  • Rolling update: actualiza sin downtime
  • Rollback: revertir en segundos
  • Es el objeto que mas vas a usar

3 replicas Auto-healing Rolling Update 


apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  strategy:
    type: RollingUpdate
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: app
        image: mi-app:v2
Deployment es el gerente. Le dices "quiero 3 pods" y el se encarga. Si uno muere lo reemplaza. Para actualizar solo cambias la version de la imagen y el hace el rolling update.

Services y Config

Services

Analogia: el recepcionista del hotel

  • Pods son efimeros — IPs cambian
  • Service da una IP estable + DNS
  • Enruta trafico a pods vivos
  • Load balancing automatico
ClusterIP NodePort LoadBalancer 

apiVersion: v1
kind: Service
metadata: { name: mi-app }
spec:
  selector: { app: mi-app }
  ports: [{port: 80, targetPort: 8000}]

Interview Tip

"Diferencia entre Service, Ingress y NodePort?"

El Service es el recepcionista. Los clientes piden "la app" y el recepcionista los enruta a un pod disponible. Sin importar cuantos pods hay o si cambian. INTERVIEW TIP (Q7): "Cual es la diferencia entre Service, Ingress y NodePort?". Service = internal phone number para un set de pods (IP estable dentro del cluster). NodePort = abre un puerto en cada nodo (30000-32767) para acceso externo via NodeIP:NodePort. LoadBalancer = pide al cloud un LB real. Ingress = L7 HTTP routing por hostname/path (necesita un Ingress Controller como nginx). TRAMPA: "Ingress es un Service type" - NO, es un objeto separado. El default Service type es ClusterIP, no NodePort. Follow-up: "Pueden coexistir NodePort e Ingress?" (Si - el Ingress tipicamente esta backed por un Service, a menudo ClusterIP, a veces NodePort en bare-metal).

ConfigMaps y Secrets

ConfigMap

Configuracion no sensible


data:
  DB_HOST: "postgres.default"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"

Misma imagen, diferentes configs para dev/prod

Secret

Datos sensibles (base64)


type: Opaque
data:
  DB_PASS: c3VwZXJzZWNyZXQ=
stringData:
  API_KEY: "live-key-123"

base64 NO es encriptacion. Habilitar encryption at rest.

Interview Tip

"Cual es la diferencia entre ConfigMap y Secret?"

Misma imagen de Docker, pero diferentes ConfigMaps para cada ambiente. Los secrets son base64 (no es encriptacion real). En produccion se usa encryption at rest. INTERVIEW TIP (Q9): "Que es un ConfigMap y que es un Secret? Por que dos?". ConfigMap = non-sensitive config (feature flags, log levels, URLs). Secret = sensitive stuff (passwords, API keys, tokens). Ambos inyectan config en pods sin bakearlo en la imagen. Usa Secrets para que la data este base64-encoded en etcd y puedas habilitar encryption at rest. TRAMPA: "Secrets son encriptados por default". NO - son base64-encoded, no encriptados. Cualquiera con `kubectl get secret -o yaml` los lee. Follow-up: "Como mantener los secrets realmente secret en produccion?" (External Secrets Operator + AWS Secrets Manager/Vault, o sealed-secrets, nunca commit a git).

Interview Tip

"Si un Pod no schedula, como debugueas?"

kubectl es como docker CLI pero para un cluster. apply despliega, get lista, describe muestra detalle, logs muestra logs, scale escala. INTERVIEW TIP (Q10): "Si un Pod no schedula, como lo debugueas?". Step 1: `kubectl describe pod ` - lee la seccion Events al final, te dira exactamente por que (ImagePullBackOff? Insufficient cpu? Tainted node?). Step 2: `kubectl get events --sort-by=.metadata.creationTimestamp` para ver el timeline. Step 3: `kubectl describe node` para ver recursos. El 80% de los failures de scheduling son: missing resources, wrong nodeSelector, o unsatisfied taints. TRAMPA: hacer `kubectl logs` en un Pending pod - no hay logs porque el container nunca arranco. Follow-up: "Cual es la diferencia entre Pending y ImagePullBackOff?" (Pending = scheduler no lo ha colocado. ImagePullBackOff = ya esta scheduled pero la imagen no se descarga).

Horizontal Pod Autoscaler

Analogia: cajas en un supermercado

  • Vertical: hacer la caja mas rapida (mas CPU/RAM)
  • Horizontal: abrir mas cajas (mas pods)
  • K8s HPA escala horizontalmente
  • Lee metricas: CPU, memoria, o custom

Si CPU > 70% → mas pods • Si baja → menos pods 


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler

Gateway API: el sucesor de Ingress

Analogía: el portero del edificio, pero con roles separados

  • Ingress (2015) = un YAML monolítico, todo en un recurso
  • Gateway API (2023, GA) = roles separados por equipo:
    • GatewayClass: el tipo de infra (proveedor)
    • Gateway: la puerta física (infra team)
    • HTTPRoute: las reglas de routing (app team)
  • Multiprotocolo: HTTP, gRPC, TCP, TLS, UDP
  • Implementaciones: Istio, NGINX, Cilium, Envoy Gateway, Traefik

# 1. Infra team crea el Gateway (la puerta fisica)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata: { name: mi-gateway }
spec:
  gatewayClassName: cilium
  listeners:
  - { name: http, port: 80, protocol: HTTP }
---
# 2. App team define SU routing
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: api-route }
spec:
  parentRefs: [{ name: mi-gateway }]
  hostnames: ["miapp.com"]
  rules:
  - matches: [{ path: { type: PathPrefix, value: /api }}]
    backendRefs: [{ name: api-svc, port: 80 }]
  - matches: [{ path: { type: PathPrefix, value: /web }}]
    backendRefs: [{ name: web-svc, port: 80 }]

Mismo caso que Ingress, pero infra y apps se mueven en recursos separados

Interview Tip

"Qué es Gateway API y por qué reemplaza a Ingress?"

Gateway API es el sucesor moderno de Ingress (GA en K8s 1.24+, 2023). Ingress mezclaba todo en un solo recurso (el operador de infra Y el app team tocaban el mismo YAML). Gateway API separa responsabilidades: GatewayClass (proveedor), Gateway (infra team decide el LB, listeners, TLS), HTTPRoute (app team define su routing sin tocar infra). Esto es CRITICO en organizaciones grandes: un cambio de routing de una app NO requiere aprobacion del infra team. Soporta mas protocolos (gRPC, TCP, TLS passthrough) y tiene mejores features de cross-namespace routing, traffic splitting, header-based routing. Si la pregunta de entrevista es sobre Ingress en 2026, mencionar Gateway API como la direccion correcta muestra que sabes lo actual. INTERVIEW TIP (Q8): "Que es Gateway API y por que reemplaza a Ingress?". Gateway API = SIG-Network successor of Ingress (networking.k8s.io/v1), GA en K8s 1.24. Separa el modelo de roles: GatewayClass (proveedor, ej cilium), Gateway (infra team), HTTPRoute (app team). Beneficios: multi-protocolo (HTTP/gRPC/TCP/TLS), expresividad (header matching, traffic split, mirror), cross-namespace routing sin tocar infra. TRAMPA: decir que Gateway API YA esta GA y es el estandar. NO decir "es experimental" o "es el futuro" - ya es el presente.
