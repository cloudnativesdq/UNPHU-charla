Paso 1

Contenedores

Empaquetando aplicaciones de forma portable

Por que contenedores?

Sin contenedores

  • "Funciona en mi maquina"
  • Dependencias conflictivas
  • Deployments inconsistentes
  • Provisioning lento (horas)

Con contenedores

  • Mismo entorno everywhere
  • Dependencias aisladas
  • Imagen inmutable
  • Provisioning en segundos
Los contenedores resuelven el problema de "funciona en mi maquina". Cada contenedor tiene su propio filesystem, red y procesos.

Dockerfile


FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
EXPOSE 8000
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

docker build -t mi-app:v1 . && docker run -p 8000:8000 mi-app:v1

Dockerfile define los pasos para construir la imagen. FROM=base, COPY=archivos, RUN=comandos, CMD=proceso principal. HEALTHCHECK es crucial para Kubernetes.

Docker Compose: Stack local


services:
  app:
    build: .
    ports: ["8000:8000"]
    depends_on:
      db: { condition: service_healthy }
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes: [pgdata:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD", "pg_isready"]
      interval: 5s

  redis:
    image: redis:7-alpine

App + Postgres + Redis con un solo docker compose up -d

Compose nos permite definir un stack multi-container localmente. Los healthchecks son importantes para el orden de arranque.

Arquitectura

Como funciona K8s

Control Plane

API Server

Puerta de entrada. API REST central. Todo comunica a traves de aqui.

etcd

Base de datos key-value distribuida. Guarda TODO el estado del cluster.

Scheduler

Asigna pods a nodos segun recursos disponibles y reglas.

Controllers

Vigilan que el estado real coincida con el estado deseado.

El API server es la puerta de entrada. etcd guarda TODO. El scheduler decide donde van los pods. Los controladores mantienen el estado deseado.

Nodos Worker

kubelet

Agente en cada nodo. Gestiona ciclo de vida de pods.

kube-proxy

Reglas de red. Implementa Services y routing.

containerd

Runtime que ejecuta los contenedores.

Cada nodo ejecuta kubelet, kube-proxy y un container runtime. Se registran con el control plane.

Paso 3

Objetos de K8s

Pods, Deployments, Services

Pods

  • 1 o mas contenedores por pod
  • Comparten misma IP y almacenamiento
  • Son efimeros: se crean y destruyen
  • En produccion: usar Deployments
  • Siempre definir resources (requests/limits)

apiVersion: v1
kind: Pod
metadata:
  name: mi-app
  labels:
    app: mi-app
spec:
  containers:
  - name: app
    image: mi-app:v1
    ports:
    - containerPort: 8000
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
Un Pod es como un host logico. Contenedores comparten red. Es efimero - si muere, se reemplaza. Por eso en produccion usamos Deployments.

Deployments


apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
  labels:
    app: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: app
        image: mi-app:v1

3 replicas Rolling Update Auto-healing Rollback

Deployment gestiona ReplicaSets que gestionan Pods. Rolling update = zero downtime. Rollback instantaneo con kubectl rollout undo.

Paso 4

Services y Config

Services

ClusterIP

Interno. DNS automatico.

NodePort

Puerto del nodo (30000+)

LoadBalancer

LB externo del cloud


apiVersion: v1
kind: Service
metadata:
  name: mi-app
spec:
  selector:
    app: mi-app
  ports:
  - port: 80
    targetPort: 8000
Services dan IP estable y DNS a pods efimeros. ClusterIP es default para comunicacion interna.

ConfigMaps y Secrets

ConfigMap

Configuracion no sensible


data:
  DB_HOST: "postgres.default"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"

Secret

Datos sensibles (base64)


type: Opaque
data:
  DB_PASS: c3VwZXJzZWNyZXQ=
stringData:
  API_KEY: "live-key-123"
Misma imagen, diferentes ConfigMaps para dev/staging/prod. Secrets son base64, no encriptacion. Habilitar encryption at rest en produccion.

Paso 5

Escalado

Horizontal Pod Autoscaler

Escala pods automaticamente segun metricas:

  • CPU utilization
  • Memoria
  • Metricas custom

replicas = ceil(current x (actual / target))


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: mi-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: mi-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
HPA lee metricas del Metrics Server. Si CPU > 70% escala up, si baja escala down con cooldown.

Paso 6

Observabilidad

Metricas + Logs + Traces

Los 3 pilares

Metricas

Prometheus

CPU, memoria, requests, latency

Logs

Loki + Promtail

Errores, stack traces, eventos

Traces

Tempo + OpenTelemetry

Flujo de requests end-to-end

Los 3 pilares de observabilidad. No se puede debuguear produccion sin ellos. Grafana unifica todo.

Prometheus

  • Pull model: scrapea endpoints /metrics
  • PromQL: lenguaje de consultas poderoso
  • Alertmanager: alertas a email/slack
  • ServiceMonitor: auto-discovery en K8s

# Requests por segundo
rate(http_requests_total[5m])

# P99 latency
histogram_quantile(0.99,
  rate(http_duration_bucket[5m]))
            

apiVersion: monitoring.coreos.com/v1
kind: Prometheus
metadata:
  name: prometheus
spec:
  serviceAccountName: prometheus
  serviceMonitorSelector:
    matchLabels:
      team: frontend
  resources:
    requests:
      memory: 400Mi
Prometheus es el estandar de facto para metricas en Kubernetes. ServiceMonitores hacen auto-discovery de pods.

Loki + Promtail


# Promtail - DaemonSet en cada nodo
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: promtail
spec:
  template:
    spec:
      containers:
      - name: promtail
        image: grafana/promtail:latest
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: containers
          mountPath: /var/lib/docker/containers
            

# Buscar errores
{app="mi-app"}
  |= "error"
  | json
  | status >= 500

# Rate de errores por pod
sum(rate({app="mi-app"}
  |= "error" [5m])) by (pod)
  • DaemonSet: un Promtail por nodo
  • LogQL: consulta como PromQL
  • Indexa labels, no texto completo
Loki es mas economico que Elasticsearch. Promtail corre como DaemonSet. LogQL permite filtrar y agregar.

Grafana: Unificando todo

Alerta en Prometheus Ver log en Loki Saltar al trace en Tempo

Grafana unifica los 3 pilares. Click en metrica te lleva al log, click en log te lleva al trace.
