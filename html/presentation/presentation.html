Contenedores

Empaquetando aplicaciones de forma portable

El problema: "Funciona en mi maquina"

Sin contenedores

  • "Funciona en mi maquina" ...pero no en produccion
  • Dependencias conflictivas entre proyectos
  • Deployments inconsistentes
  • Onboarding lento (instalar todo desde cero)

Con contenedores

  • Mismo entorno en dev, staging y produccion
  • Dependencias aisladas entre proyectos
  • Imagen inmutable y versionada
  • Onboarding en minutos: docker compose up
Pregunta: cuantos han tenido el problema de instalar algo y que no funcione? Ese es el problema que Docker resuelve.

VMs vs Contenedores

Virtual Machines

Cada VM tiene su propio SO completo
Pesado. Gigabytes. Lento de arrancar. 


+-----+  +-----+  +-----+
| App |  | App |  | App |
+-----+  +-----+  +-----+
| Libs|  | Libs|  | Libs|
+-----+  +-----+  +-----+
|Guest|  |Guest|  |Guest|  <- SO completo!
| OS  |  | OS  |  | OS  |
+=====+==+=====+==+=====+
|     Hypervisor        |
+=======================+
|    Host Operating OS  |
+=======================+

Contenedores

Comparten el kernel del host
Ligero. Megabytes. Arranca en segundos. 


+-----+  +-----+  +-----+
| App |  | App |  | App |
+-----+  +-----+  +-----+
| Libs|  | Libs|  | Libs|
+=====+==+=====+==+=====+
|   Container Engine    |
+=======================+
|    Host Operating OS  |  <- 1 solo SO!
+=======================+
La diferencia clave: VMs incluyen un SO completo por instancia (GBs). Contenedores comparten el kernel del host (MBs). Un contenedor arranca en milisegundos vs minutos para una VM.

Imagen vs Contenedor

Imagen (Image)

Es la receta

  • Inmutable (no cambia)
  • Plantilla de solo lectura
  • Se versiona con tags
  • Vive en un registry (Docker Hub)

Contenedor (Container)

Es el lunchbox

  • Instancia viva de una imagen
  • Tiene su proceso, red y storage
  • Se crea con docker run
  • Puedes tener N contenedores de 1 imagen

Imagen = clase  •  Contenedor = objeto

La analogia de programacion orientada a objetos: la imagen es la clase, el contenedor es la instancia. O la receta (imagen) vs el pastel ya horneado (contenedor).

Docker Hub y Registries

  • Docker Hub = GitHub para imagenes
  • Miles de imagenes oficiales: nginx, postgres, python, redis
  • docker pull descarga una imagen
  • docker push sube tu imagen
  • Empresas usan registries privados (ECR, GCR, Harbor)

# Descargar imagen oficial
docker pull python:3.12-slim

# Ver imagenes locales
docker images

# Subir tu imagen
docker tag mi-app usuario/mi-app:v1
docker push usuario/mi-app:v1
Docker Hub es como un GitHub pero para imagenes de contenedores. Las imagenes oficiales son mantenidas y seguras.

Dockerfile: La receta


FROM python:3.12-slim         # Imagen base

WORKDIR /app                  # Directorio de trabajo
COPY requirements.txt .       # Copiar dependencias
RUN pip install -r requirements.txt  # Instalar

COPY . .                      # Copiar codigo fuente
EXPOSE 8000                   # Puerto expuesto
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

docker build -t mi-app:v1 . && docker run -p 8000:8000 mi-app:v1

Explica cada instruccion: FROM=base, WORKDIR=donde trabajamos, COPY=traer archivos, RUN=ejecutar al construir, EXPOSE=documentar puerto, CMD=que corre el contenedor. HEALTHCHECK es crucial para K8s despues.

Comandos esenciales de Docker


# Construir imagen desde un Dockerfile
docker build -t mi-app:v1 .

# Ejecutar un contenedor
docker run -d --name api -p 8000:8000 mi-app:v1

# Ver contenedores corriendo
docker ps

# Ver logs del contenedor
docker logs api

# Entrar a un contenedor en vivo
docker exec -it api bash

# Parar y limpiar
docker stop api && docker rm api

Estos 5 comandos son el 80% del uso diario de Docker

Si solo recuerdan 5 comandos de esta charla, que sean estos. build, run, ps, logs, exec. Con eso ya pueden trabajar.

Docker Compose: Stack multi-container

  • Una app real tiene multiples servicios
  • App + Base de datos + Cache
  • Un archivo YAML define TODO
  • docker compose up -d y listo
  • Ideal para desarrollo local

services:
  app:
    build: .
    ports: ["8000:8000"]
    depends_on:
      db: { condition: service_healthy }

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD", "pg_isready"]

  redis:
    image: redis:7-alpine
Compose nos permite levantar un stack completo con un solo comando. Los healthchecks garantizan que la app no arranque antes que la DB este lista.

Kubernetes

Orquestacion de contenedores a escala

Por que Kubernetes?

Docker solo

  • Si un contenedor muere — se queda muerto
  • Escalar manualmente (crear mas contenedores a mano)
  • Sin balanceo de carga automatico
  • Actualizar sin downtime — muy dificil
  • Si el servidor se cae, todo se cae

Con Kubernetes

  • Auto-healing: reemplaza pods muertos
  • Auto-scaling: escala segun demanda
  • Service discovery + load balancing
  • Rolling updates con rollback instantaneo
  • Si un nodo se cae, K8s reubica los pods
Docker es genial para una app en un servidor. Pero que pasa cuando tienes 50 apps en 10 servidores y uno se cae? Kubernetes resuelve eso.

Arquitectura: Control Plane

El cerebro del cluster — toma las decisiones

API Server

Puerta de entrada. Todo pasa por aqui. kubectl habla con este.

etcd

Base de datos key-value. Guarda TODO el estado del cluster.

Scheduler

Decide en que nodo va cada pod segun recursos y reglas.

Controllers

Bucle constante: estado deseado vs estado real. Lo corrige si hay diferencia.

El control plane es el cerebro. API server es la puerta. etcd guarda todo. El scheduler decide donde van las cosas. Los controllers mantienen el estado deseado.

Arquitectura: Nodos Worker

Los musculos del cluster — ejecutan los contenedores

kubelet

El capataz. Recibe ordenes del API Server. Gestiona pods en su nodo.

kube-proxy

Reglas de red. Implementa Services y balanceo de carga.

containerd

El runtime. Ejecuta los contenedores (Docker, containerd, CRI-O).

Los worker nodes hacen el trabajo pesado. Cada uno tiene kubelet (capataz), kube-proxy (red) y un container runtime. Se registran con el control plane.

Objetos de K8s

Pods, Deployments, Services

Pods

Analogia: roommates compartiendo apartamento

  • 1 o mas contenedores por pod
  • Comparten misma IP y almacenamiento
  • Efimeros: nacen y mueren
  • Nunca se reubican — si mueren, se crean nuevos
  • En produccion: usar Deployments

apiVersion: v1
kind: Pod
metadata:
  name: mi-app
  labels:
    app: mi-app
spec:
  containers:
  - name: app
    image: mi-app:v1
    ports:
    - containerPort: 8000
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
Un Pod es como un apartamento donde viven 1 o mas roommates (contenedores). Comparten todo (red, storage). Son efimeros. Por eso usamos Deployments para gestionarlos.

Deployments

Analogia: un gerente que contrata/despide

  • Define cuantos pods quieres (replicas)
  • Si un pod muere — crea uno nuevo
  • Rolling update: actualiza sin downtime
  • Rollback: revertir en segundos
  • Es el objeto que mas vas a usar

3 replicas Auto-healing Rolling Update 


apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  strategy:
    type: RollingUpdate
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: app
        image: mi-app:v2
Deployment es el gerente. Le dices "quiero 3 pods" y el se encarga. Si uno muere lo reemplaza. Para actualizar solo cambias la version de la imagen y el hace el rolling update.

Services y Config

Services

Analogia: el recepcionista de un hotel

  • Los pods son efimeros — IPs cambian
  • Service da una IP estable + DNS
  • Enruta trafico automaticamente a pods vivos
  • Load balancing incluido

ClusterIP — interno

NodePort — puerto del nodo

LoadBalancer — LB del cloud


apiVersion: v1
kind: Service
metadata:
  name: mi-app
spec:
  selector:
    app: mi-app
  ports:
  - port: 80
    targetPort: 8000
El Service es el recepcionista. Los clientes piden "la app" y el recepcionista los enruta a un pod disponible. Sin importar cuantos pods hay o si cambian.

ConfigMaps y Secrets

ConfigMap

Configuracion no sensible


data:
  DB_HOST: "postgres.default"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"

Misma imagen, diferentes configs para dev/prod

Secret

Datos sensibles (base64)


type: Opaque
data:
  DB_PASS: c3VwZXJzZWNyZXQ=
stringData:
  API_KEY: "live-key-123"

base64 NO es encriptacion. Habilitar encryption at rest.

Misma imagen de Docker, pero diferentes ConfigMaps para cada ambiente. Los secrets son base64 (no es encriptacion real). En produccion se usa encryption at rest.

# Desplegar una app

Escalado

Horizontal Pod Autoscaler

Analogia: cajas en un supermercado

  • Vertical: hacer la caja mas rapida (mas CPU/RAM)
  • Horizontal: abrir mas cajas (mas pods)
  • K8s HPA escala horizontalmente
  • Lee metricas: CPU, memoria, o custom

Si CPU > 70% → mas pods • Si baja → menos pods 


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: mi-app-hpa
spec:
  scaleTargetRef:
    kind: Deployment
    name: mi-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
HPA es como el supervisor del supermercado que abre mas cajas cuando hay mucha fila. Escala de 2 a 10 pods segun la demanda. Todo automatico.

Observabilidad

Metricas + Logs + Traces

Por que observabilidad?

Sin observabilidad

Conducir de noche, sin luces, sin tablero

  • "Algo se cayo pero no se que"
  • Debuguear a ciegas en produccion
  • Usuarios reportan antes que tu
  • MTTR alto (horas o dias)

Con observabilidad

Tablero completo + mecanico diagnostico

  • Deteccion proactiva de problemas
  • Root cause en minutos
  • Alertas antes de que el usuario note
  • MTTR bajo (minutos)
Observabilidad es como el tablero de tu carro. Sin el, vas a ciegas. No puedes arreglar lo que no puedes ver. MTTR = Mean Time To Recovery.

Los 3 pilares

Metricas

Prometheus

Numeros en tiempo real: CPU, memoria, requests, latencia

Logs

Loki + Promtail

Errores, stack traces, eventos: que paso exactamente

Traces

Tempo + OpenTelemetry

Flujo de un request end-to-end: donde se tardo

Los 3 pilares. Metricas te dicen ALGO esta mal. Logs te dicen QUE esta mal. Traces te dicen DONDE. Grafana unifica los tres.

Prometheus: Metricas

  • Pull model: scrapea /metrics
  • PromQL: lenguaje de consultas
  • Alertmanager: alertas a email/slack
  • ServiceMonitor: auto-discovery en K8s

# Requests por segundo
rate(http_requests_total[5m])

# P99 latency
histogram_quantile(0.99,
  rate(http_duration_bucket[5m]))
            

apiVersion: monitoring.coreos.com/v1
kind: Prometheus
metadata:
  name: prometheus
spec:
  serviceAccountName: prometheus
  serviceMonitorSelector:
    matchLabels:
      team: frontend
  resources:
    requests:
      memory: 400Mi
Prometheus es el estandar de facto. En vez de que la app envie metricas (push), Prometheus las pide (pull) cada 15-30 segundos.

Loki + Promtail: Logs


# Promtail - DaemonSet en cada nodo
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: promtail
spec:
  template:
    spec:
      containers:
      - name: promtail
        image: grafana/promtail:latest
        volumeMounts:
        - name: varlog
          mountPath: /var/log
            

# Buscar errores
{app="mi-app"}
  |= "error"
  | json
  | status >= 500

# Rate de errores por pod
sum(rate({app="mi-app"}
  |= "error" [5m])) by (pod)
  • Loki = como Elasticsearch pero mas barato
  • Indexa labels, no texto completo
  • LogQL: filtra como PromQL
Loki es la alternativa economica a ELK/Elasticsearch. Promtail corre en cada nodo (DaemonSet) y envia logs a Loki.

Grafana: Unificando todo

Metricas

Prometheus datasource

Logs

Loki datasource

Traces

Alerta en Prometheus Click → ver log en Loki Click → saltar al trace en Tempo

Grafana es la interfaz visual. Un click en una metrica te lleva al log relacionado. Un click en el log te lleva al trace. Todo conectado.
