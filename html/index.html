PASO 1

Contenedores

Empaquetando aplicaciones de forma portable

Por que contenedores?

Mismo hardware, mucho menos overhead

⚠ Máquina Virtual

VS

✓ Contenedores

Los contenedores resuelven el problema de "funciona en mi maquina". Cada contenedor tiene su propio filesystem, red y procesos.

Dockerfile


FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .
EXPOSE 8000
HEALTHCHECK CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
          

docker build -t mi-app:v1 .
docker run -p 8000:8000 mi-app:v1
Dockerfile define los pasos para construir la imagen. FROM = base, COPY = archivos, RUN = comandos, CMD = proceso principal. HEALTHCHECK es crucial para Kubernetes.

PASO 2

Docker Compose

Multi-container en tu laptop

App Demo: Acortador de URLs

Flujo de una petición de extremo a extremo

Esta es la app demo: un acortador de URLs construido con 3 servicios. El browser hace una petición GET con el código corto. Nginx en el Frontend sirve la SPA y hace proxy al Backend. El Backend (Flask) consulta Redis para resolver el código a la URL original. Redis almacena pares clave-valor: url:{code} → URL larga. Finalmente el Backend responde con un 301 Redirect a la URL original. Todo esto corre en la misma Docker network — los servicios se comunican por nombre.

Stack local completo


services:
  app:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD", "pg_isready"]
      interval: 5s

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:
Compose nos permite definir un stack multi-container localmente. App + Postgres + Redis con un solo docker compose up. Los healthchecks son importantes para el orden de arranque.

Arquitectura

Dos partes: Control Plane (el cerebro) y Nodos (los trabajadores). Todo se comunica via API Server. etcd guarda TODO el estado.

Pods

Unidad minima. Comparten red y storage.


apiVersion: v1
kind: Pod
metadata:
  name: mi-app
  labels:
    app: mi-app
spec:
  containers:
  - name: app
    image: mi-app:v1
    ports:
    - containerPort: 8000
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "500m"
  • 1 o mas contenedores por pod
  • Comparten misma IP
  • Efimeros: no crear directamente
  • resources: siempre definir requests y limits

En produccion: usar Deployments, nunca Pods solos.

Un Pod es como un host logico. Los contenedores dentro comparten red. Siempre definir resources - sin esto, el scheduler no puede tomar buenas decisiones.

Deployments


apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
  labels:
    app: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: app
        image: mi-app:v1
        ports:
        - containerPort: 8000

3 replicas Rolling Update Auto-healing Rollback: kubectl rollout undo

Deployment gestiona ReplicaSets, que gestionan Pods. Si un Pod muere, se recrea. Rolling update = zero downtime. Si algo sale mal, rollback instantaneo.

Services

ClusterIP

Interno. DNS: mi-app.default.svc

NodePort

Puerto del nodo (30000+)

LoadBalancer

LB externo del cloud


apiVersion: v1
kind: Service
metadata:
  name: mi-app
spec:
  selector:
    app: mi-app
  ports:
  - port: 80
    targetPort: 8000
Services dan una IP estable y DNS a pods efimeros. ClusterIP es el default y suficiente para comunicacion interna. LoadBalancer para exponer al mundo exterior.

Configuracion

ConfigMaps, Secrets y Namespaces

ConfigMap

Config no sensible


apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: "postgres.default"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"

Secret

Datos sensibles (base64)


apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  DB_PASS: c3VwZXJzZWNyZXQ=
stringData:
  API_KEY: "live-key-123"
ConfigMaps y Secrets separan configuracion del codigo. Misma imagen Docker, diferentes ConfigMaps para dev/staging/prod. Secrets son base64, no encriptacion. Habilitar encryption at rest en produccion.

Namespaces


kubectl create namespace produccion
kubectl apply -f app.yaml -n produccion
kubectl get all -n produccion
  • Aislamiento logico dentro del cluster
  • Cuotas de recursos (ResourceQuota)
  • Politicas de red (NetworkPolicy)
  • Defaults: default, kube-system
Namespaces son como "clusters virtuales". Separar por ambiente o equipo.

PASO 5

Escalado

De 3 replicas a auto-scaling inteligente

Horizontal Pod Autoscaler


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: mi-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: mi-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

replicas = ceil(current × (actual / target)) — escala automaticamente

HPA lee metricas del Metrics Server y ajusta replicas. Si CPU > 70%, escala up. Si baja, escala down con cooldown.

Los 3 pilares de observabilidad. No se puede debuguear produccion sin ellos. Prometheus para metricas, Loki para logs, Tempo para traces. Grafana unifica todo en dashboards.

Prometheus

Metricas con pull-based scraping


apiVersion: monitoring.coreos.com/v1
kind: Prometheus
metadata:
  name: prometheus
spec:
  serviceAccountName: prometheus
  serviceMonitorSelector:
    matchLabels:
      team: frontend
  resources:
    requests:
      memory: 400Mi
  enableAdminAPI: false
  • Pull model: scrapea endpoints /metrics
  • PromQL: lenguaje de consultas
  • Alertmanager: alertas por email/slack
  • ServiceMonitor: auto-discovery en K8s
  • Time series DB: almacenamiento local

# Requests por segundo
rate(http_requests_total[5m])

# P99 latency
histogram_quantile(0.99,
  rate(http_duration_seconds_bucket[5m]))
Prometheus es el estandar de facto para metricas en Kubernetes. ServiceMonitores hacen auto-discovery de pods con labels. PromQL es poderoso para consultas y alertas.

Loki + Promtail

Logs agregados, como Prometheus pero para logs


# Promtail - recoge logs de cada nodo
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: promtail
spec:
  selector:
    matchLabels:
      app: promtail
  template:
    spec:
      containers:
      - name: promtail
        image: grafana/promtail:latest
        args:
        - -config.file=/etc/promtail/config.yml
        volumeMounts:
        - name: varlog
          mountPath: /var/log
        - name: containers
          mountPath: /var/lib/docker/containers
              

# LogQL - busca errores
{app="mi-app"}
  |= "error"
  | json
  | status >= 500

# Rate de errores por servicio
sum(rate({app="mi-app"}
  |= "error" [5m])) by (pod)
  • DaemonSet: un Promtail por nodo
  • Recoge logs de /var/log y containers
  • LogQL: consulta como PromQL
  • Indexa labels, no el texto completo
Loki es mas economico que Elasticsearch porque solo indexa labels. Promtail corre como DaemonSet en cada nodo para recoger logs. LogQL permite filtrar y agregar logs como PromQL.

Tempo + OpenTelemetry

Distributed tracing end-to-end

OpenTelemetry

  • Instrumentacion automatica
  • SDK para Python, Go, Java, Node...
  • Exporta traces a cualquier backend
  • W3C Trace Context standard

Tempo

  • Backend de traces de Grafana
  • Compatible con Jaeger, Zipkin
  • Busca traces por traceID
  • Conecta metricas + logs + traces

from opentelemetry import trace
from opentelemetry.sdk.trace \
  import TracerProvider

trace.set_tracer_provider(
  TracerProvider()
)
tracer = trace.get_tracer("mi-app")

@app.get("/api/data")
async def get_data():
  with tracer.start_as_current_span(
    "get_data"
  ) as span:
    span.set_attribute(
      "http.method", "GET"
    )
    data = fetch_from_db()
    return data
OpenTelemetry es el estandar abierto para instrumentacion. Tempo guarda los traces y los conecta con metricas y logs via traceID. Asi puedes saltar de una alerta en Grafana al log especifico y al trace exacto.

Grafana: Unificando todo

Los 3 pilares → un solo panel de control

Grafana unifica los 3 pilares. Click en una metrica te lleva al log. Click en el log te lleva al trace. Existe desde un solo dashboard. Esto es observabilidad real.
