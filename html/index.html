El Problema

Antes de Kubernetes... como escalabamos?

Imaginemos que tenemos una aplicacion que crece. De 1 servidor a 10, a 100, a 1000. Como los gestionamos? Como aseguramos que esten saludables? Como hacemos updates sin downtime?

El Problema

Sin Orquestacion

  • Escalado manual
  • Sin auto-healing
  • Deployments riesgosos
  • Configuracion inconsistente
  • Sin balanceo de carga

Con Orquestacion

  • Escalado automatico
  • Auto-healing nativo
  • Rolling updates
  • Configuracion declarativa
  • Service discovery + LB
La diferencia fundamental es pasar de gestion manual a declarativa. Tu declaras el estado deseado y Kubernetes se encarga de mantenerlo.

Que es Kubernetes?

Kubernetes (K8s) es una plataforma de codigo abierto para
automatizar el despliegue, escalado y gestion
de aplicaciones en contenedores.

Portable

On-premise, cloud, hibrido

Extensible

CRDs, operators, plugins

Auto-sanable

Reinicio, reemplazo, salud

Originalmente desarrollado por Google, basado en su sistema interno Borg. Ahora es mantenido por la Cloud Native Computing Foundation (CNCF). El nombre "K8s" viene de las 8 letras entre la K y la s.

Historia

2003-2015
Borg / Omega — Sistemas internos de Google para gestion de contenedores a escala masiva
2014
Kubernetes — Google libera el codigo como open source
2015
v1.0 — Primera release estable. Donado a la CNCF
2016-2018
Adopcion masiva — AWS EKS, Azure AKS, GCP GKE
2026
v1.32+ — Estandar de la industria, usado por millones de organizaciones
Google ejecutaba billones de contenedores por semana con Borg. Kubernetes es la evolucion de esa experiencia, hecha open source y democratizada.

Arquitectura de Kubernetes

Kubernetes sigue un modelo cliente-servidor con un plano de control y nodos workers. El plano de control toma decisiones globales, los nodos ejecutan las cargas de trabajo.

Control Plane

kube-apiserver

API REST central. Todo comunica a traves de aqui.

etcd

Base de datos key-value distribuida. Estado del cluster.

kube-scheduler

Asigna pods a nodos segun recursos y reglas.

kube-controller-manager

Controladores: ReplicaSet, Deployment, Node, etc.

El API server es la puerta de entrada. etcd guarda TODO el estado. El scheduler decide donde van los pods. Los controladores vigilan que el estado real coincida con el deseado.

Nodos (Workers)

kubelet

Agente en cada nodo. Gestiona el ciclo de vida de los pods.

kube-proxy

Mantiene reglas de red. Implementa Services.

Container Runtime

containerd, CRI-O. Ejecuta los contenedores.

Cada nodo ejecuta kubelet (gestion de pods), kube-proxy (red) y un container runtime. Los nodos se registran con el control plane para recibir instrucciones.

Pods

La unidad mas pequena y simple en Kubernetes

  • Contiene uno o mas contenedores
  • Comparten red (misma IP) y almacenamiento
  • Son efimeros: se crean y destruyen
  • Nunca se crean directamente en produccion

apiVersion: v1
kind: Pod
metadata:
  name: mi-pod
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
Un Pod es como un "host logico" con sus contenedores compartiendo recursos. Es efimero - si muere, se reemplaza. Por eso en produccion usamos Deployments.

Deployments

Gestion declarativa de Pods y ReplicaSets

Deployments


apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80

3 replicas Rolling Update Auto-healing

Deployment es el objeto que usamos en produccion. Define cuantas replicas queremos, que imagen usar, y como actualizar. Si un pod muere, el Deployment crea uno nuevo automaticamente.

Estrategias de Update

Rolling Update

Actualiza gradualmente los pods sin downtime. Es el default.


spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0

Recreate

Elimina todos los pods viejos antes de crear nuevos. Hay downtime.


spec:
  strategy:
    type: Recreate
Rolling Update es ideal para produccion: los pods nuevos se crean antes de eliminar los viejos. Recreate es util cuando la app no soporta versiones concurrentes.

Services

Abstraccion estable para acceder a un grupo de Pods

ClusterIP

IP interna. Solo accesible dentro del cluster. Default.

NodePort

Expone en un puerto de cada nodo (30000-32767).

LoadBalancer

Crea un LB externo en el cloud provider.


apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
Los pods son efimeros y cambian de IP. Services proveen una IP estable y DNS. El tipo LoadBalancer integra directamente con el cloud provider (AWS ELB, GCP LB, etc).

ConfigMaps y Secrets

ConfigMap

Configuracion no sensible


apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_HOST: "db.internal"
  LOG_LEVEL: "info"

Secret

Datos sensibles (base64)


apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  DB_PASSWORD: c3VwZXJzZWNyZXQ=

Separa configuracion del codigo → misma imagen, diferentes ambientes

ConfigMaps y Secrets permiten externalizar la configuracion. Secrets estan codificados en base64 (no encriptados por defecto). Para produccion, habilitar encryption at rest en etcd.

kubectl

La navaja suiza para interactuar con Kubernetes

Comandos Esenciales


# Ver recursos del cluster
kubectl get nodes
kubectl get pods -A
kubectl get deployments

# Describir un recurso en detalle
kubectl describe pod mi-pod

# Aplicar manifiestos
kubectl apply -f deployment.yaml

# Escalar un deployment
kubectl scale deployment nginx --replicas=5

# Ver logs en tiempo real
kubectl logs -f deployment/nginx

# Ejecutar comando en un pod
kubectl exec -it mi-pod -- /bin/bash

# Port-forward para debugging local
kubectl port-forward svc/nginx 8080:80
Estos son los comandos que usaran el 90% del tiempo. kubectl get y describe son los mas importantes para empezar. apply es declarativo - define el estado deseado y kubectl lo aplica.

Escalado en Kubernetes

Horizontal Pod Autoscaler (HPA)

El HPA escala pods automaticamente segun metricas:

  • CPU utilization
  • Memoria
  • Metricas custom (requests/sec, queue depth)

Formula: replicas deseadas = ceil(replicas actuales × (metrica actual / metrica objetivo)) 


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deployment
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
El HPA revisa periodicamente las metricas y ajusta las replicas. La formula basicamente dice: si el uso de CPU supera el 70%, escala hacia arriba. Si baja, escala hacia abajo. Tiene un periodo de estabilizacion para evitar flapping.

Namespaces

Aislamiento logico dentro de un cluster


# Listar namespaces
kubectl get namespaces

# Crear namespace
kubectl create namespace produccion

# Aplicar en un namespace
kubectl apply -f app.yaml -n produccion

# Ver pods en un namespace
kubectl get pods -n produccion
  • Aisla recursos entre equipos/proyectos
  • Cuotas de recursos por namespace
  • Políticas de red entre namespaces
  • Namespaces default: default, kube-system, kube-public
Namespaces son como "clusters virtuales" dentro del cluster fisico. Utiles para separar dev/staging/prod o diferentes equipos. kube-system contiene los componentes del sistema.
